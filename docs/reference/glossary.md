# localpass Technical Glossary

This glossary defines the core cryptographic, security, architectural, and user interface terms used throughout the localpass codebase. 

---

### Cryptographic & Security Terms

#### Vault
The encrypted persistence layer for the password manager, stored physically as a `.nlk` file inside the Windows `%APPDATA%/Roaming/localpass` directory. The vault contains the encrypted JSON payload representing all stored credentials, secure notes, and passkeys.
- **Physical Location on Windows:** `C:\Users\<Username>\AppData\Roaming\localpass\vault.nlk`
- **File Format:** JSON envelope containing Base64-encoded ciphertext, salt, nonce, and cryptographic hashes (see [vault-file-format.md](../api/vault-file-format.md) for binary layouts).

#### Master Password
The primary user-supplied secret passphrase. It is never stored on disk, never sent over the network, and never retained in plaintext memory. It is used exclusively as the input to a memory-hard key derivation function (KDF) to derive the vault encryption key.

#### Derived Key
A 256-bit (32-byte) symmetric AES encryption key produced by processing the Master Password and the vault's unique salt through the Argon2id key derivation function. This key is used to decrypt and encrypt the vault payload. It is temporarily held in a secure, zeroable bytearray in the Python process memory and is zeroed out immediately when the vault locks.

#### Salt
A cryptographically secure, random 16-byte value generated via `os.urandom(16)` when the vault is first initialized. It is stored in plaintext in the vault's JSON envelope. The salt ensures that two identical master passwords do not produce the same derived key, rendering pre-computed rainbow tables and multi-vault brute-force attacks impossible.

#### Nonce
A cryptographically secure, random 12-byte initialization vector generated via `os.urandom(12)` uniquely for every single vault write operation. In the AES-256-GCM authenticated encryption scheme, reusing a nonce (number used once) with the same key catastrophically compromises the security of the cipher.

#### GCM Tag
A 16-byte authenticated tag produced by the AES-256-GCM cipher during encryption. It serves as an integrity proof, confirming that the encrypted ciphertext has not been modified, corrupted, or tampered with. In the Python `cryptography` library, the tag is automatically appended to the ciphertext during encryption and verified during decryption.

#### Session Token
A cryptographically secure, random 32-byte hexadecimal string (64 characters) generated by the Python daemon on a successful handshake. The extension service worker caches this token strictly in non-persistent memory. The token is transmitted inside the custom `X-NL-Token` header on every HTTP request and verified in constant time. It has a default time-to-live (TTL) of 28,800 seconds (8 hours) and is immediately invalidated upon locking.

#### Zero Knowledge
The core security paradigm of localpass. Plaintext credentials, keys, and metadata are processed, encrypted, and decrypted entirely on the local client machine. The software holds no capability to recover data without the Master Password, and no decrypted data or secrets are ever transmitted outside the security boundary of the local machine.

#### Argon2id
A memory-hard, highly secure key derivation function (defined in RFC 9106) designed to resist GPU/ASIC-accelerated brute-force attacks. localpass uses the Argon2id variant which combines data-independent and data-dependent memory accesses to mitigate side-channel timing attacks.
- **Default Parameters:** Time cost = 3 iterations, Memory cost = 65,536 KB (64 MB), Parallelism = 2 threads, Hash length = 32 bytes.

#### AES-256-GCM
Advanced Encryption Standard operating in Galois/Counter Mode with a 256-bit key. It is an Authenticated Encryption with Associated Data (AEAD) block cipher that provides both confidentiality and high-integrity assurance by producing an authentication tag along with the ciphertext.

#### FIDO2
A set of specifications (comprising WebAuthn and CTAP2) enabling strong, cryptographic passwordless authentication. localpass implements FIDO2-compliant virtual authenticator capabilities to register, store, sign, and manage hardware-agnostic passkeys.

### Browser Extension Architecture

#### Content Script
A JavaScript file injected by the browser extension into the context of every visited webpage (`matches: ["<all_urls>"]` running at `document_idle` or `document_start`). Content scripts observe page updates, detect login/password fields, inject user-autofill buttons, and render dropdown option panels.
- **Project Files:** `localpass-extension/content.js`, `utils/detector.js`, `utils/filler.js`, `utils/webauthn_interceptor.js`.

#### Service Worker
The background script (`localpass-extension/background.js`) running in a separate, persistent execution context inside the extension process. It manages session state, coordinates communications between popup/content scripts and the Python local server, and intercepts WebAuthn requests.

#### Popup Context
The short-lived JavaScript execution environment associated with the extension's user interface panel (`popup.html` and `popup.js`). It is initialized when the user clicks the extension toolbar icon and is completely destroyed when the popup is closed.

#### Bridge
The unified messaging function `sendBg()` implemented in both `popup.js` and `content.js`. It wraps `chrome.runtime.sendMessage` to route runtime messages asynchronously to the background Service Worker and returns promises.

#### Handshake
The initial challenge-response token exchange protocol executed between the extension Service Worker and the local Python server to establish a secure session. The extension generates a 32-byte hex challenge nonce, and the server signs it using an HMAC-SHA256 keyed with the newly issued Session Token, proving its authenticity before the token is trusted.

#### NL Button
A small, circular, interactive cyan button containing the localpass padlock SVG icon. It is dynamically injected inside or directly next to detected input fields (username, email, or password) on the active webpage. Clicking it activates the inline dropdown autocomplete box.

#### Inline Dropdown
A sleek, framework-agnostic floating autocomplete suggestion panel rendered below a focused input field. It displays matched credentials (title, username, favicons) and passkeys retrieved from the local vault, letting the user autofill credentials with a single click.

#### Domain Matching
The algorithms implemented in `localpass/core/domain_trust.py` and `background.js` to find stored vault credentials relevant to the active webpage. The algorithm normalizes URLs, extracts the registered domain using a Public Suffix List (PSL)-aware parser, and calculates a trust score to prioritize exact matches and prevent phishing.

#### MutationObserver
A native browser API used by `content.js` to observe additions, deletions, and structural modifications within the webpage's DOM. This ensures that dynamically loaded login fields, single-page application forms, and modal fields are immediately injected with NL buttons.

#### registerView
A core router registration function `registerView(name, fn)` defined in `popup.js`. It registers visual views (e.g. `entries`, `search`, `generator`, `settings`, `detail`) as single-page application (SPA) renderers, allowing seamless navigation.

#### viewStack
A navigation history array `viewStack` maintained dynamically in `popup.js`. It functions as a LIFO stack to support browser-popup routing, tracking the chronological order of visited views to enable backwards navigation via `navigateBack()`.
